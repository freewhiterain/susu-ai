一、背景

我们公司想给员工做一个内部 AI 助手「小苏」。员工在钉钉 / 飞书 / 企业微信里就能跟它聊天，管理员通过一个 Web 管理后台维护知识库和查看对话日志。它能干这些事：

  • 员工在 IM 里 @ 它问「我能休几天年假？」→ 它从公司员工手册里找答案，并附上原文出处
  • 员工问「张三今天来上班了吗？」→ 它去查考勤系统 API，告诉你结果
  • 员工问「上周三的销售额是多少？」→ 它查订单系统 API，并算出汇总
  • 员工说「再帮我总结一下」→ 它记得上一轮聊了什么
  • 管理员在 Web 后台上传 / 删除文档，看到员工今天问了哪些问题、哪些没答上来

你这次的任务，就是把「小苏」做出来，放到一个公开仓库（GitHub 或 Gitee 都行，方便我们下载考核），并提供一个能让我们当场体验的 Demo。

二、你要做的：必做清单

功能 1：文档知识库 权重 12%

  • 上传文档：支持 Markdown / PDF / Word / TXT 至少 3 种格式
  • 文档列表：能看到所有已上传文档、索引状态（pending / indexed / failed）
  • 删除文档：删除后该文档不再参与问答
  • 增量更新：上传同名文件能识别并替换，不重复处理 分水岭

功能 2：智能问答 权重 18%

  • 基于知识库的问答：答案必须带引用（哪个文件、哪一段原文），实现方式不限
  • 多轮对话：能记住上下文，支持「再详细讲讲」「换种方式说」
  • 流式输出：边生成边显示，不要憋一坨再返回
  • 拒答机制：检索不到相关内容时，明确说「文档里没找到」，不准瞎编
  • 来源定位：引用要能点击跳转 / 高亮原文位置（不只是文件名）

功能 3：工具调用 权重 12%

至少接入 2 个工具，模型要能自主决定什么时候查文档、什么时候调工具：

  • 工具 A：内部 API（必做） —— 你自己用 FastAPI / Express / Next.js API 起一个 mock 服务，提供至少 2 个接口（如 /api/employee/{id}、/api/attendance、/api/orders），数据自己 mock 几条 JSON 即可
  • 工具 B：通用工具 —— 计算器、当前时间、单位换算，任选其一即可

⚠️ 关键考点：模型要根据问题自己判断用哪个工具，不是写死的 if-else。

功能 4：IM 集成 权重 15% 核心

这是「小苏」最终给员工用的入口。从 钉钉 / 飞书 / 企业微信 中任选一个接入，多做一个加分（见加分项）：

  • 注册成机器人：员工能在群里 @ 小苏，或私聊小苏
  • 消息收发：接收用户消息，调用后端处理后回复
  • 多轮对话：在 IM 里也要保持上下文（按用户 ID + 会话维度区分）
  • 引用展示：答案的引用来源要在 IM 里清晰呈现（卡片 / Markdown / 链接均可）
  • 错误友好：后端崩了 / 模型超时，IM 里要有兜底回复，不能用户那边一直转圈

⚠️ 这一项不能跳过。如果只交了 Web 没接 IM，视为未完成核心需求。

小提示：钉钉机器人和飞书机器人都有免费的开发者沙箱，企业微信需要企业认证（可以用试用账号）。建议优先选你之前接触过的，或者钉钉（文档最完善）。

功能 5：Web 管理后台 权重 8%

给管理员用的，不是给终端员工用的。终端员工在 IM 里聊天，不在这里聊。

  • 文档管理：上传 / 列表 / 删除（即功能 1 的 UI 入口）
  • 对话日志：能看到所有员工的提问历史、回答内容、是否触发了工具、Token 消耗
  • 设置（可简单）：切换模型、查看接入的 IM 状态
  • 备用对话页（可选）：保留一个 Web 端聊天页，方便管理员调试和你面试时演示

不要求好看，但要求能用。可以用 shadcn/ui、Ant Design、Element Plus 等现成组件库。

功能 6：工程化 分水岭

多数人会丢分，请认真对待：

  • 一条命令启动：docker-compose up / pnpm dev / ./scripts/start.sh
  • 配置全部外置：API Key、模型名、Base URL 走环境变量；提供 .env.example
  • 日志输出到 logs/ 目录，不要只打 console
  • 错误兜底：API 超时 / 限流 / 模型胡说，要重试或降级，不能直接 500
  • 至少 3 条自动化测试，其中至少 1 条用 Mock LLM，不依赖真实 API
  • README 完整：项目介绍、架构图（手画也行）、安装、使用、技术栈、Roadmap
  • .env 不能进仓库（提交了直接淘汰）

三、加分项

不强制，但能拉开差距，面试时会重点关注：

  • 🌟 多端 IM 接入：钉钉 / 飞书 / 企业微信中接通 ≥ 2 个，且**消息处理逻辑能复用**（不是各写一份）
  • 🌟 富消息形态：IM 里支持卡片消息、文件上传问答、@ 群成员等高级交互
  • 🌟 多模型适配：通过统一接口同时支持 ≥ 2 家供应商
  • 🌟 Token 计数与成本展示：UI 里能看到这次对话花了多少 Token / 多少钱
  • 🌟 MCP：把「小苏」做成一个 MCP server，能被 Claude Desktop / Cursor 调用
  • 🌟 可观测性：接 Langfuse / OpenTelemetry，能看到每次请求的链路
  • 🌟 Evals：写一份自动化评测脚本，至少 20 条 case，能跑出准确率

四、数据怎么准备

所有数据让 AI 自己生成即可，不需要你去外面找资料。建议规模如下：

知识库文档（5-10 篇，约 1-3MB）

  • 员工手册.md：年假、报销、考勤、加班、出差等公司规章制度
  • 新人入职指南.md：账号开通、设备申领、培训流程等
  • FAQ.md：员工常问的几十条问题
  • 再随意补几篇 PDF / Word / TXT 测试格式兼容

直接让 Claude / GPT 帮你生成这些文档，10 分钟搞定。

Mock API 数据（每个 JSON 文件 10-30 条即可）

// employees.json
[{"id": "001", "name": "张三", "dept": "研发部", "level": "P5"}, ...]

// attendance.json
[{"emp_id": "001", "date": "2026-06-10", "check_in": "09:15", "check_out": "18:30"}, ...]

// orders.json
[{"id": "O20260601", "amount": 1280, "date": "2026-06-01", "customer": "A 公司"}, ...]

同样让 AI 一次性生成，注意覆盖一些边界情况（迟到、加班、退款订单等）方便测试。

我们不考你找数据的能力，考的是你用这些数据搭出一个能跑的 AI 应用。所以请把时间花在刀刃上。

五、技术栈：完全自由（但有底线）

完全自由：前端、后端语言、向量数据库、模型供应商。

底线（不遵守扣分）：

  • Python 用 uv 管理依赖，虚拟环境必须叫 .venv，禁止 pip install
  • Node 用 pnpm，禁止 commonjs
  • 用 Next.js → v15.4+；用 React → v19+；用 Tailwind → v4+
  • 启动 / 测试 / 部署的命令统一放在 scripts/*.sh
  • 数据结构强类型（Python 用 Pydantic / dataclass，TS 别滥用 any）
  • 单文件 ≤ 500 行（Python/TS）/ 800 行（Go/Rust），单目录 ≤ 8 个文件

这些不是刁难，是入职后的工程规范。能做到说明你能融入。

六、必交的文档

6.1 README.md（中文）

项目介绍、截图、架构图、安装、使用、技术栈、Roadmap

6.2 AI_USAGE.md 最重要 · 单独评分

告诉我们你怎么用 AI 工具做这个项目的，至少回答下面 5 个问题：

  1. 你用了哪些 AI 工具（Claude Code / Cursor / Codex / 其他）？分别用在哪些环节？
  2. 举一个具体的例子：你给 AI 的 prompt 是什么、它生成的代码哪里能用、哪里你必须改、为什么改？
  3. 有没有一次 AI 把你带沟里去的经历？你怎么发现的、怎么收拾的？
  4. 你怎么验证 AI 生成的代码是对的？
  5. 如果让你再做一遍，你会怎么调整 AI 的使用方式？

⚠️ 简历里说「精通 Claude Code」的人很多，能把这五个问题答得有血有肉的人很少。全是 AI 自动生成的官话直接出局。

6.3 自评.md（不超过 1 页）

  • 你做的是什么、解决谁的什么问题
  • 你最得意的 1 个技术决策 + 1 个被迫的妥协
  • 如果再给你 1 周，你会优先做什么

七、面试时怎么验收

我们会用一份标准测试问题清单测你的项目（你做的时候就可以拿这个清单自测）。所有对话题目都在 IM 里跑（你接入的钉钉 / 飞书 / 企业微信任意一个），第 7.6 项检查 Web 管理后台。

7.1 基础问答（看文档问答是否工作）

  1. 「员工每年有几天年假？」（预期：从员工手册命中，带引用）
  2. 「报销发票需要什么材料？」（预期：命中员工手册）
  3. 「新人入职第一天要做哪些事？」（预期：从新人入职指南命中）

7.2 工具调用（看 Agent 是否会选工具）

  1. 「员工 001 是哪个部门的？」（预期：调 /api/employee/001）
  2. 「上周一共多少订单？」（预期：调 /api/orders + 自己算）
  3. 「现在几点？」（预期：调时间工具）

7.3 多轮对话（看上下文是否保持）

  1. 接着 Q4 问：「他上周来上班几天？」（预期：能理解「他」=员工 001，再调考勤 API。注意：在 IM 里要按用户/会话维度隔离上下文，不能 A 问的话被 B 接着了）

7.4 拒答与边界（看会不会瞎编）

  1. 「我们公司 CEO 的家庭住址是？」（预期：拒答，不准瞎编）
  2. 「2030 年的销售目标是多少？」（预期：说不知道）

7.5 工程鲁棒性

  1. 把模型 API Key 改成无效值，看 IM 里的错误回复是否友好（不能用户那边一直转圈或永久无响应）

7.6 Web 管理后台检查

  • 能看到上面 1-10 题在 IM 里发生的所有对话日志（谁问的、问的什么、回答了什么、调了哪些工具）
  • 从后台上传一个新文档，立刻去 IM 里问相关问题，能命中
  • 从后台删除一个文档，再去 IM 里问，不应再命中

面试时：我们会让你现场在 IM 里跑这些问题（你提前把面试官加进测试群即可），并随机挑 1-2 个让你改一处代码（比如让你新增一个工具、或调整 prompt），看你是不是真的写过、能不能维护。

十、常见问题

Q：IM 平台必须三个都接吗？

不用。钉钉 / 飞书 / 企业微信任选一个即可，做到能稳定收发消息、保持上下文、回复带引用就算合格。多接一个加分（见加分项）。

建议优先选你之前用过的；都没用过的话，钉钉文档最完善、注册门槛最低，可以从它开始。

Q：没有公司账号能注册机器人吗？

能。三个平台都有个人开发者沙箱 / 试用版，用个人手机号就能注册一个测试组织、创建机器人。具体流程查官方文档：

  • 钉钉：开放平台 → 开发者后台 → 创建企业内部应用 / 机器人
  • 飞书：开放平台 → 开发者后台 → 创建企业自建应用
  • 企业微信：要求企业认证，可申请试用组织

Q：Web 端能不能不做？

不能完全不做。Web 管理后台是必交项（文档管理 + 对话日志），但权重不高。

Web 端的聊天页面是可选的——我们鼓励你保留一个简单的聊天页方便管理员调试和你面试时演示，但如果时间紧张可以不做，最终验收以 IM 里的对话为准。

Q：可以用 v0 / bolt / lovable 起手吗？

可以，但最终代码要是你能解释、能改、能维护的。面试会让你现场改一处。

Q：可以参考开源项目吗？

可以，但必须在 README 顶部明确标注 fork 自哪里、你改了什么。我们关心的是你的增量价值。隐瞒来源 = 失格。

Q：做完会不会被白嫖？

不会。代码归你自己，要不要加 LICENSE、加哪种你自己定。哪怕没来我们公司，这也是简历里能放的项目，我们鼓励你长期维护它。

Q：可以多人合作吗？

不行，个人笔试。
